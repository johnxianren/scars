**2. Approaches considered, in order**
1. **Memoize `quote()` wholesale (the sprint-41 TODO in core.py)** — considered first because the code pointed at it; **rejected without implementing**. `quote()` is not pure: COMPLIANCE.md §4.2 requires a complete fresh audit trail per issuance including byte-identical retry baskets, and `test_every_issuance_is_fully_audited` enforces it. Memoizing the whole quote would either drop audit records (compliance violation) or, if audits were still emitted per call, save almost nothing since the cost is the table build, not the 8-line arithmetic. Also mechanically awkward: baskets are lists and `Catalog` is deliberately unhashable.
2. **Cache the discount table keyed on `(id(catalog), version)` in a module-level dict** — considered, **abandoned** for id-reuse-after-GC risk and unbounded growth; storing the cache on the `Catalog` instance itself is safer and self-cleaning.
3. **Cache the table on the `Catalog` instance, invalidated by `catalog.version`** — **implemented**. The loader bumps `version` on every items/rules change (documented contract in models.py; `test_quote_reflects_rule_reload` exercises it). Cache read uses `getattr(..., None)` so pre-existing live `Catalog` objects surviving a hot code reload don't crash.
4. **Speed up the cold build itself** so the first quote after a reload can't blow p99 — **implemented**: bucket tiers by category once (a tier only matches items of its own category), sort each bucket by rate descending, and stop each SKU's scan at the first match (max-rate-wins makes the first match the winner). Cold build: ~49 ms → ~1.8 ms. Verified the optimized table is exactly equal to the old naive max-over-all-tiers result on the full fixture catalog (200/200 SKUs).
5. **2D dominance/interval data structures for tier matching** — briefly considered as a further cold-build optimization, **abandoned** as overkill once the bucketed scan measured at ~1.8 ms.
